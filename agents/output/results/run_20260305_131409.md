## 6) Evaluation Reasoning Highlights and Suggestions

**Root causes identified (from evaluation_reasoning)**

- **Not blocking / matching:**
  - Blocking pair completeness ‚âà **0.99**.
  - Matching F1 ‚âà **0.88‚Äì0.91**.
  - Cluster analysis totally clean (1‚Äì1 matches, no large clusters).
  - Conclusion: entity linkage quality is high.

- **Primary issues are in fusion + evaluation alignment:**

  1. **Coverage / missing fused records**
     - Several gold IDs have `fused_id = NaN` and `reason = 'missing_fused_value'` for all attributes.
     - Likely because `DataFusionEngine.run(..., include_singletons=False)` drops unmatched entities.
     - Result: false negatives at the cluster level; lowers accuracy on multiple attributes per record.

  2. **Genres representation**
     - Gold: single flattened set of atomic genres.
     - Fused: list of 1‚Äì3 large comma-separated strings (one per source).
     - Evaluation uses `set_equality_match` on list-like columns ‚Üí nearly all genre comparisons fail (accuracy **0.05**).
     - Root cause: `union`-style fusion applied to **strings** that should have been tokenized lists.

  3. **Author / title / publisher normalization**
     - Authors: extra tags like `(Goodreads Author)`, roles `(Editor)`, `(Illustrator)`, multiple contributors.
     - Titles: trailing series/edition markers in parentheses.
     - Publishers: imprints, divisions, HTML `&amp;`, longer descriptive names.
     - Fusion strategy uses **`longest_string`** for these, systematically selecting the noisiest variant; evaluation expects a cleaner canonical string.

  4. **Numeric attributes fused as strings**
     - `publish_year`, `page_count`, `rating`, `numratings`, `price` all fused via `longest_string`.
     - Evaluation uses numeric-aware functions like `numeric_tolerance_match` and `year_only_match`.
     - Type/aggregation mismatch can cause avoidable errors (e.g., 464 vs 480 page counts, slight year discrepancies).

**Key suggestions for next iteration**

1. **Improve coverage**
   - Run fusion with singletons enabled:
     - `include_singletons=True` in `DataFusionEngine.run`.
   - This ensures unmatched gold entities appear as singleton fused records and can be evaluated.

2. **Fix genres fusion**
   - Implement a **custom genres fuser** that:
     - Splits comma-separated strings into atomic tags.
     - Merges and deduplicates across all sources.
     - Returns a single sorted list or comma-joined string of unique genres.
   - Alternatively, normalize `genres` into lists before fusion, then use a true set-union fuser.
   - Expected impact: `genres_accuracy` should increase from **0.05** to near-perfect.

3. **Replace `longest_string` with semantics-aware fusers**
   - **Author:**
     - Strip parenthetical role/platform tags and secondary authors.
     - Majority vote on normalized author name.
   - **Title:**
     - Strip trailing series/edition parentheses.
     - Choose shortest normalized title or majority vote.
   - **Publisher:**
     - HTML-decode (`&amp;` ‚Üí `&`), strip divisions in parentheses.
     - Majority vote on normalized name.
   - **Language:**
     - Simple normalized majority vote string fuser.

4. **Use numeric aggregators for numeric fields**
   - For `publish_year`, `page_count`, `rating`, `numratings`, `price`:
     - Parse numerics and apply a median/majority fuser rather than string length.
     - Aligns with numeric tolerance checks in evaluation.

5. **Align list normalization across pipeline**
   - Reuse the same `detect_list_like_columns` / `normalize_list_like_columns` logic **before fusion** for attributes like `genres`.
   - Ensures that the fused representation is compatible with the evaluation‚Äôs `set_equality_match`.

**Overall guidance**

- Keep current blocking/matching configurations; they are already strong.
- Focus improvements on:
  - **Fusion functions** (authors, titles, publishers, numeric fields, genres).
  - **Coverage** (include singletons).
  - **Representation alignment** with evaluation rules (especially for list-like and numeric attributes).
