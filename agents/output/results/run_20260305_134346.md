## 6) Evaluation Reasoning Highlights and Suggestions

**Diagnosed dominant problems:**

- **Not blocking/matching quality** (pairwise F1 is high, blocking recall ~0.99), but:
  1. **Over-merged, ambiguous clusters**:
     - Large hubs where many distinct books share author/publisher/year.
     - One-to-many ratios 0.05–0.37, big max cluster sizes (up to 90).
  2. **Fusion strategy too naive and cluster-agnostic**:
     - `longest_string` for text chooses noisy or wrong values (titles, authors, publishers).
     - `maximum` for numerics (years, page counts) over-weights outliers/longest editions.
     - `genres` union over stringified lists causes structural mismatch with evaluation.
  3. **Coverage gaps / missing fused rows**:
     - `include_singletons=False` in fusion likely drops unmatched records.
     - `build_eval_view` maps each fused cluster to a single eval ID; some gold IDs in `_fusion_sources` never get their own aligned row → `missing_fused_value`.
  4. **Representation mismatches**:
     - Authors: role suffixes and multi-author strings not normalized (e.g., `(Goodreads Author)`, `(Editor)`).
     - Genres: fused as list of long strings instead of list of atomic genre tokens expected by evaluator.

**Key suggested changes (fusion & evaluation focused):**

1. **Increase coverage:**
   - Run fusion with `include_singletons=True` so every input record appears in fused output.
   - Change evaluation view construction to:
     - Emit **one eval row per gold ID present in `_fusion_sources`**, not a single representative per cluster.

2. **Improve text fusers (title/author/publisher/language):**
   - Replace `longest_string` with **cleaned, frequency-based fusers**:
     - Strip suffixes and roles from authors (e.g., `(Goodreads Author)`, `(Editor)`).
     - Normalize publishers (`&amp;` → `&`, remove legal suffixes like `Inc.`).
     - For titles, remove trailing series/edition parentheses and choose the **most frequent normalized core title**, preferring shorter, less noisy versions.
   - Optionally use **source preference** (e.g., prioritize Amazon or Goodreads for `title`).

3. **Improve numeric fusers (publish_year, page_count, etc.):**
   - Replace `maximum` with:
     - **Min or median** for `publish_year` (avoid future/reprint years).
     - **Median (or trimmed median)** for `page_count`, `rating`, `numratings`, `price`.
   - Optionally drop clear numeric outliers (>20–30% away from median) before fusing.

4. **Fix genres fusion and representation:**
   - Normalize `genres` in all base datasets to **lists of atomic tokens** (split by commas, deduplicate).
   - Use a fuser that performs a **set union over tokens** with a reasonable cap on size.
   - Ensure evaluation normalization sees the same atomic-token lists, so `set_equality_match` works; this should lift `genres` accuracy from 0.

5. **Cluster-aware / ambiguity-aware fusion:**
   - For large or high-degree clusters (e.g., size > 5 or with high ambiguous ratio):
     - Use **more conservative fusion**:
       - Prefer titles/authors from a single trusted source.
       - Restrict numeric and genre fusion to records whose titles are highly similar.
     - Optional advanced step: split clusters by **title-consistency** (e.g., high Jaro–Winkler) and fuse each subcluster separately.

6. **Evaluation-side tolerance improvements (optional but easy):**
   - When comparing `author` and `publisher`, apply the same normalization used in fusion (strip roles, normalize case) before `tokenized_match`.
   - This avoids penalizing harmless formatting differences such as `Vintage Books USA` vs `Vintage`.

**Priority order for next run:**

1. Enable `include_singletons=True` and fix eval ID alignment to reduce `missing_fused_value`.
2. Swap out `longest_string`/`maximum` for majority/median-based fusers, with simple normalization for author/publisher/title.
3. Normalize and refactor `genres` fusion to operate on atomic tokens.
4. Add cluster-size-aware conservative fusion rules for large/hub clusters.
