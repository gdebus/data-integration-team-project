Data Integration & LLMs

Using LLM-Agents in Data Integration Pipelines

Authors: Onur Can Memis, Gregor Debus, Abdullah Momin, Nicolas Blazek

University of Mannheim, Data Integration & LLMs

Introduction

Data integration, the process of combining records from multiple, independently created data sources into a single, unified view, is one of the oldest and most labour-intensive problems in data management. A typical integration pipeline must align heterogeneous schemas, identify which records across sources refer to the same real-world entity, and then fuse conflicting attribute values into a single canonical record. Each of these steps traditionally requires significant manual effort: domain experts write blocking rules, tune matching thresholds, and hand-craft fusion strategies for every new combination of datasets.

This project investigates whether Large Language Models (LLMs) can substantially automate that process. Rather than using an LLM as a simple classifier or feature extractor, we embed it as the decision-making core of an autonomous agent that generates, executes, evaluates, and iteratively improves a complete integration pipeline. The agent receives raw datasets, entity matching test sets, and a validation set as input and produces a fused output table as output, with no manual intervention required for blocking key selection, comparator tuning, or resolver configuration.

When accuracy falls short, the agent diagnoses the root cause and decides which pipeline stage to revisit, a process we call investigation-driven iteration.

Project Overview

We developed a series of eight agent variants (Agents I through VIII, plus the final Agent VII), each extending the previous with additional capabilities. This incremental design allows us to measure the contribution of each component in isolation:

  • Agent I establishes the baseline: an LLM generates a complete pipeline from dataset profiles alone, with no pre-computed strategy information.
  • Agent II adds automated blocking and matching testers that empirically determine optimal strategies before code generation.
  • Agents III-IV introduce a reasoning node that analyses evaluation failures and provides targeted improvement suggestions (rule-based and ML-based matching variants respectively).
  • Agents V-VI add cluster analysis and a documentation search tool that gives the LLM access to PyDI’s API reference at generation time.
  • Agent VIII is a bridge version that introduces the first investigation orchestrator and normalization node, proving the value of these concepts but relying on hardcoded routing logic.
  • Agent VII is a ground-up rebuild that replaces passive reasoning with an active investigation agent capable of writing and executing diagnostic code, introduces mechanical code guardrails, a declarative normalization system, per-run output isolation, regression protection, and a reworked cluster analysis that rates all five PyDI post-clustering algorithms.

Core Goals

  • Automation: reduce manual intervention in mapping and merging data to zero: the agent should produce a fused table from raw inputs without human-written rules.
  • Accuracy: leverage LLM reasoning to achieve fusion accuracy above 80% on the validation set across all use cases, competitive with hand-tuned baselines.
  • Scalability: design a pipeline architecture that handles diverse datasets, from small restaurant directories to large game catalogues with hundreds of thousands of candidate pairs.

Architecture

Our data integration system is built as a modular, agent-based pipeline using LangGraph state graphs. Each agent version extends the previous one with additional capabilities, enabling us to evaluate the incremental value of each component. All agents share a common core (schema matching, dataset profiling, LLM-driven pipeline generation, code execution, and evaluation) but differ in the upstream analysis stages and the feedback loops they support.

Shared State

Every agent variant operates on the same AgentState typed dictionary that is threaded through the LangGraph graph. Key fields include the list of input dataset paths, dataset profiles, the generated pipeline code, execution results and attempt counters, evaluation code and metrics, and for the more advanced agents blocking configurations, matching configurations, schema correspondences, and the matcher mode (rule_based or ml). Because the state is shared, each node can read the outputs of all prior nodes and every re-entry into the generation node carries the full history of errors and evaluation feedback.

Common Nodes

Several processing nodes appear across all or most agent variants. They form the backbone of the pipeline and are described here once to avoid repetition.

Match Schemas

The Match Schemas node aligns column names across the input datasets. It invokes an LLM-based schema matching routine that receives the raw column lists from every source and produces a dictionary of correspondences. These correspondences are persisted in the state and consumed by all downstream nodes so that blocking, matching, and fusion operate on a unified attribute namespace. If correspondences are already present in the state (e.g. from a previous invocation), the node is skipped.

Profile Datasets

The Profile Datasets node uses an LLM tool call to invoke the ProfileDatasetTool on every input dataset. The tool returns a JSON profile containing row counts, null percentages per column, and column data types. The profiles are saved to disk and stored in the agent state so that the pipeline generation node can reason about data characteristics such as sparsity and type mismatches.

Generate Pipeline

The Generate Pipeline node is the core LLM code-generation step. It assembles a large system prompt that includes an example integration pipeline using the PyDI library. It consumes the list of dataset paths, first-row previews, dataset profiles, etc. It also conditionally includes schema correspondences and evaluation reasoning from prior iterations. The LLM is asked to produce a complete Python script that performs blocking, entity matching, and data fusion.

On the first invocation the prompt simply says "Create the integration pipeline." On subsequent invocations it includes the previously generated code together with any execution errors or evaluation feedback, asking the LLM to fix or improve the script.

Execute Pipeline

The Execute Pipeline node runs the generated Python script as a subprocess. If the process exits with a non-zero return code, the stderr output is captured and fed back to the Generate Pipeline node. The agent retries up to a maximum number of attempts before terminating. A successful execution transitions the workflow to the evaluation branch.

Generate Evaluation Code & Execute Evaluation

After a successful pipeline run the agent generates evaluation code following a similar pattern: a system prompt that includes an example evaluation script, the generated pipeline code, the fused output path, and a preview of the gold-standard test set. The generated evaluation script loads the fused output and the test set, computes per-column and overall accuracy, and writes structured metrics to and output file. The evaluation code is then executed as a subprocess with the same retry logic as the pipeline execution step.

Evaluation Decision

The Evaluation Decision node reads the JSON metrics produced by the evaluation script. It checks two termination criteria: either the overall fusion accuracy exceeds a threshold (85%) or the maximum number of evaluation attempts (three) has been reached. If neither condition is met, the workflow continues to the next stage, which, depending on the agent variant, is either termination or the reasoning node.

Agent I: Basic Pipeline Creation (RB)

Agent I is the simplest variant and establishes the baseline. It supports only rule-based matching and does not include any upstream analysis for blocking or matching strategy selection. Its workflow is a straight-line sequence with two conditional retry loops.

Workflow

  1. Profile Datasets: characterise each input dataset.
  2. Generate Pipeline: the LLM receives the example pipeline, dataset profiles, and first-row previews, and produces a complete integration script. No blocking or matching configuration is provided; the LLM must infer suitable strategies from the data alone.
  3. Execute Pipeline: run the generated script. On failure, loop back to Generate Pipeline with the error message (max. 3 attempts).
  4. Generate Evaluation Code: create an evaluation script for the fused output.
  5. Execute Evaluation: run the evaluation. On failure, loop back to Generate Evaluation Code (max. 3 attempts).
  6. Evaluation Decision: if overall accuracy ≥ 85% or attempts ≥ 3, terminate.

Because Agent I gives the LLM no pre-computed strategy information, the quality of the generated pipeline depends entirely on how well the model can infer reasonable blocking keys and matching rules from the dataset profiles and the example code. This makes Agent I a useful lower bound for comparison.

Agent II: Blocking & Matching Tester (RB)

Agent II extends Agent I by inserting two dedicated evaluation nodes before pipeline generation: the Blocking Tester and the Matching Tester. Both nodes run automated experiments to determine the optimal strategy for each dataset pair and persist the results in the agent state. Like Agent I, Agent II supports only rule-based matching.

Blocking Tester

The Blocking Tester receives the schema-aligned datasets and the entity-matching test sets. For each dataset pair it evaluates multiple blocking strategies: Standard Blocker (exact equality), Token Blocker (shared tokens), Sorted Neighbourhood Blocker (sliding window), and Embedding Blocker (semantic similarity via sentence transformers). Each strategy is scored by its Pairs Completeness (PC), which measures the proportion of true matches retained after blocking. The best strategy per pair is selected subject to a PC threshold (default 0.9) and a maximum candidate count. The resulting blocking configuration is a JSON object specifying, for each pair, the blocker type, the blocking columns, and the relevant parameters (e.g. top_k for the Embedding Blocker or window for the Sorted Neighbourhood Blocker).

Matching Tester

The Matching Tester operates on the candidate pairs produced by the optimal blocking strategy. It uses the LLM to propose comparator configurations (StringComparator, NumericComparator, DateComparator) with associated weights and thresholds for a rule-based matcher. Multiple configurations are evaluated against the test sets and scored by F1. The best configuration per pair is retained in a matching configuration JSON that includes comparator types, column mappings, weights, preprocessing steps, and the decision threshold. The tester iterates up to a configurable number of attempts, feeding evaluation results back to the LLM to refine the configuration.

Workflow Extension

  1. Match Schemas
  2. Profile Datasets
  3. Blocking Tester: determine optimal blocking strategy per pair.
  4. Matching Tester: determine optimal matching strategy per pair.
  5. Generate Pipeline: now receives blocking and matching configurations as additional context in the system prompt.
  6. Execute → Evaluate → Decide (same as Agent I).

By providing pre-computed, empirically validated blocking and matching configurations, Agent II removes much of the guesswork from the pipeline generation step. The LLM can focus on correctly translating the configuration into executable code rather than also having to invent a strategy.

Agents III & IV: Reasoning (RB & ML)

Agents III and IV introduce a Reasoning Node that closes the feedback loop between evaluation and pipeline generation. After the Evaluation Decision determines that accuracy is below the threshold, instead of simply re-running the generation node with the raw metrics, the workflow first passes through the Reasoning Node which performs a structured analysis of why the pipeline underperformed.

Agent III uses a rule-based matcher, while Agent IV uses a machine-learning-based matcher. The ML variant changes the example pipeline, the matching tester configuration, and the prompts to use feature extraction, classifier training, and MLBasedMatcher instead of weighted comparators and RuleBasedMatcher.

Reasoning Node

The Reasoning Node receives the aggregate evaluation metrics, a sample of detailed per-record debugging events from the fusion evaluation (up to 50 JSONL records), the current pipeline code, the evaluation code, and the blocking and matching configurations. It is prompted to identify dominant error types (false positives, false negatives, missed blocks, bad matches) and to determine whether the errors originate in blocking, matching, schema alignment, preprocessing, or the fusion logic. The node produces concise, structured improvement suggestions that are injected into the next pipeline generation prompt as additional context.

Because the blocking and matching strategies have already been independently validated by the tester nodes, the Reasoning Node is instructed to focus its suggestions on the fusion stage e.g., choosing different attribute fusers, adjusting trust strategies, or fixing preprocessing inconsistencies that cause column mismatches during evaluation.

ML Matcher Mode (Agent IV)

When the matcher mode is set to ml, several components adapt automatically. The Matching Tester proposes comparator features for a FeatureExtractor rather than weights for a rule-based decision function. The generated pipeline trains a binary classifier on the labelled test set using the extracted features and applies it via MLBasedMatcher. The evaluation and reasoning nodes operate identically; only the matching strategy and example code differ.

Workflow Extension

  1. Match Schemas → Profile → Blocking Tester → Matching Tester (same as Agent II).
  2. Generate Pipeline → Execute Pipeline (with retry loop).
  3. Generate Evaluation → Execute Evaluation (with retry loop).
  4. Evaluation Decision: if accuracy < 95% and attempts < 3, continue to Reasoning Node; otherwise terminate.
  5. Reasoning Node: analyse errors, produce improvement suggestions.
  6. Loop back to Generate Pipeline: the suggestions are appended to the system prompt, enabling the LLM to make targeted corrections in the next iteration.

Agents V & VI: Cluster Analysis & Documentation Search Tool

Agents V and VI represent the most complete version of the system. They retain all capabilities of Agents III/IV and add two new components: a Cluster Analysis node and a Search Documentation Tool. Agent V uses rule-based matching; Agent VI uses ML-based matching.

Cluster Analysis

After the pipeline produces entity correspondences, the Cluster Analysis node examines the cluster distribution of the matched entities. It computes cluster sizes, identifies the frequency of each cluster size, and derives diagnostic ratios: the small-cluster ratio (fraction of clusters of size 2), the large-cluster ratio (fraction above a threshold), and the one-to-many ratio. A high one-to-many ratio or the presence of very large clusters indicates over-matching, which the node diagnoses and addresses by recommending a strategy e.g., AdjustMatchingConfig with a higher threshold.

The cluster analysis is fed into subsequent iterations of the pipeline generation so that the LLM can tighten matching thresholds or adjust blocking parameters to reduce spurious clusters. This proved especially valuable for the Books use case, where editions of the same title (e.g. multiple printings of The Fellowship of the Ring) created clusters of size 9–30.

Search Documentation Tool

The Search Documentation Tool gives the pipeline generation node access to the PyDI library documentation at generation time. When enabled, the agent issues targeted queries to retrieve documentation for specific classes and functions e.g., the parameters of EmbeddingBlocker, the available fuser functions in DataFusionEngine, or the correct usage of RuleBasedMatcher comparators. The retrieved documentation snippets are appended to the system prompt, reducing hallucinated API calls and improving first-attempt success rates.

Enabling the documentation tool increases token usage considerably (since documentation text is injected into the prompt), so it is controlled by a global flag (INCLUDE_DOCS) and used selectively.

Workflow Extension

  1. Match Schemas → Profile → Blocking Tester → Matching Tester.
  2. Generate Pipeline: now additionally augmented with documentation search results when INCLUDE_DOCS is enabled.
  3. Execute Pipeline.
  4. Cluster Analysis: diagnose the entity cluster distribution.
  5. Generate Evaluation → Execute Evaluation.
  6. Evaluation Decision → Reasoning Node (conditional loop).

Agent VIII: Investigation & Normalization (Bridge Version)

Agent VIII represents a transitional iteration between the notebook-based Agents I-VI and the fully rebuilt Agent VII. It introduced several of the ideas that would later be refined in Agent VII, most notably an investigation system, a normalization node, and the beginnings of modular helper files, but retained the monolithic pipeline structure and many of the limitations of its predecessors.

The primary goal of Agent VIII was to close the gap between evaluation feedback and pipeline improvement. Where Agents III-VI had a single-shot Reasoning Node that returned free text, Agent VIII introduced an investigator orchestrator that combined multiple LLM calls (diagnostics, reasoning, normalization assessment) with a deterministic routing score to decide which pipeline stage to revisit. It also added a normalization node capable of transforming source data before matching, a capability entirely absent in earlier agents.

What It Introduced

  • Investigation orchestrator: a multi-step process that called three separate LLMs (integration diagnostics, evaluation reasoning, normalization assessment) and combined their outputs with a weighted scoring formula to route the pipeline to normalization, matching, or pipeline generation.
  • Normalization node: an LLM-generated custom Python transform applied to source datasets before matching. Unlike Agent VII’s PyDI-backed NormalizationSpec, Agent VIII’s normalization wrote ad-hoc code, making it flexible but prone to runtime failures.
  • Helper file restructuring: the first extraction of logic from the monolithic notebook into dedicated Python modules (investigator_orchestrator.py, investigator_routing.py, investigation_helpers.py).
  • Fusion size monitoring: an estimator that predicted expected fusion output row counts, enabling detection of anomalous over- or under-matching.
  • Probe system (early version): initial diagnostic probes that provided structured evidence to the investigation LLM, though the investigator could not execute code to verify hypotheses.

Key Limitations

While Agent VIII proved the value of investigation-driven routing and normalization, it had several shortcomings that motivated the complete rebuild in Agent VII:

  • Hardcoded routing logic: the investigator used a weighted scoring formula with 10+ hand-tuned parameters (e.g., llm_base=0.45, stagnation_penalty=0.14) rather than letting the LLM decide. This made routing brittle and difficult to tune across different use cases.
  • Three separate LLM calls: diagnostics, reasoning, and normalization assessment were each a standalone LLM invocation, which was expensive and could produce contradictory recommendations.
  • No code execution: the investigation LLM could only reason passively about evidence. It could not write and execute diagnostic Python to verify hypotheses before making a decision.
  • Unstable normalization: because normalization used LLM-generated Python code rather than a declarative spec, a single syntax error or type mismatch could crash the entire pipeline.
  • No code guardrails: LLM-generated pipeline code was executed as-is, with no mechanical fixes for common mistakes like invalid list_strategy values, missing imports, or wrong output paths.
  • No regression protection: a new pipeline iteration that regressed from 60% to 20% accuracy was accepted without question.
  • Keyword-based normalization detection: the system used string matching (scanning for words like “normaliz” or “canonical”) to decide whether normalization was needed, which was fragile and missed many real normalization opportunities.

Agent VIII served as an important proving ground: it demonstrated that investigation-driven routing and data normalization could meaningfully improve fusion quality, but also revealed that hardcoded decision logic and ad-hoc code generation needed to be replaced with LLM-driven decisions and mechanical guardrails. These lessons directly shaped the architecture of Agent VII.

Agent VII: Autonomous Investigation Pipeline

Agent VII is a ground-up rebuild of the agent pipeline. Where Agents I-VI incrementally added nodes to a shared notebook architecture, Agent VII restructures the entire system into a modular Python codebase with dedicated helper modules, an LLM-driven investigation agent that can execute diagnostic code, a comprehensive probe system, mechanical code guardrails, and run-scoped output isolation. The result is a substantially more capable and stable agent that achieves validation accuracy above 80% across all five use cases (companies, games, music, restaurants, books).

The motivation for this redesign stems from fundamental limitations observed in Agents I-VI. Those earlier agents suffered from three recurring problems. First, the single-shot Reasoning Node could only speculate about what went wrong: it received aggregate metrics and a handful of debug events, but could never actually inspect the fused data, trace a mismatched value back to its source, or verify a hypothesis with code. Second, the feedback loop was a blunt instrument: no matter what the problem was (bad blocking, wrong fusion resolver, format mismatch, over-matching), the system always routed back to “regenerate the entire pipeline,” forcing the LLM to rewrite hundreds of lines of working code just to change a single fusion strategy. Third, LLM-generated code was executed verbatim, so common mistakes (invalid list_strategy arguments, missing imports, wrong output paths) would crash the pipeline and consume an entire retry cycle before anyone discovered the problem.

Agent VII addresses each of these issues with a dedicated subsystem. The investigation agent replaces passive reasoning with active, multi-turn diagnosis that can write and execute Python code against the actual data. The routing system replaces the single feedback loop with five distinct targets (normalization, blocking tester, matching tester, pipeline generation, and human review), allowing the agent to fix precisely the stage that is broken. The code guardrails system mechanically corrects known LLM mistakes before execution, so that runtime failures reflect genuine data problems rather than trivial code errors. Together with the probe system, the pipeline scaffold, and regression protection, these innovations transform the agent from a trial-and-error code generator into an empirical investigator that diagnoses root causes and applies targeted fixes.

Key Innovation

Agent VII replaces the single-shot Reasoning Node of Agents III-VI with a multi-turn investigation agent that receives pre-computed evidence from 11 diagnostic probes, can write and execute Python code to test hypotheses, and then makes an intelligent routing decision about which pipeline stage to fix next. Only three minimal safety overrides exist; all other routing decisions are made by the LLM based on evidence.

Design Principles

  1. LLM-first: the agent leverages LLM intelligence for decisions. Deterministic code provides evidence; the LLM interprets and decides. For example, the 11 diagnostic probes compute structured metrics about mismatch reasons, source trustworthiness, and cluster health, but it is the investigation LLM that interprets this evidence in context and decides whether the root cause is a normalization gap, a wrong fusion resolver, or an upstream matching problem.
  2. Advisory, not prescriptive: prompts present evidence and options. The LLM decides strategy based on the specific data. The cluster tester, for instance, rates all five post-clustering algorithms as “recommended,” “viable,” or “unsuitable” and presents sample problematic clusters, but the final choice of algorithm is left to the investigation LLM, which can weigh the evidence against the broader pipeline context.
  3. Active investigation: the investigation LLM can write and execute diagnostic Python to test hypotheses before deciding; it does not just reason passively. This is the single biggest improvement over Agents III-VI: instead of guessing from aggregate metrics, the investigator can load the actual fused DataFrame, compare specific values against expectations, parse metadata columns to trace a fused value back to its source dataset, and confirm or reject a hypothesis before committing to a routing decision.
  4. Evidence-driven: every recommendation is backed by concrete metrics, mismatch samples, or probe outputs. The source attribution probe, for example, does not merely report that the “label” attribute has low accuracy; it traces each fused label back to its source dataset, computes per-source match rates against the validation set, and recommends a specific resolver with a concrete trust ordering.
  5. Mechanical guardrails: critical fixes (imports, trust maps, numeric coercion, sub-column stripping, ML post-clustering) are enforced mechanically rather than relying on LLM compliance with prompt instructions. Prompt instructions are inherently unreliable: an LLM may ignore them, misinterpret them, or forget them across iterations. The guardrails ensure that certain categories of known errors are impossible regardless of what the LLM generates.
  6. Minimal hardcoded overrides: only structural invalidity, budget exhaustion, and rejection streaks override the LLM’s routing decision. Every other routing choice is made by the investigation LLM based on evidence, preserving the system’s ability to adapt to novel data patterns that a hardcoded rule system would not anticipate.
  7. Full traceability: investigation transcripts, iteration history, and detailed logging are saved for post-hoc analysis. Every investigation turn, including the diagnostic code the LLM wrote, its stdout/stderr output, and the LLM’s interpretation, is recorded in a JSON transcript, making it possible to audit exactly why the agent made each decision.

Graph Structure & Routing

Agent VII implements a LangGraph state machine in pipeline_agent.py (~2080 lines). The SimpleModelAgent class defines the graph, all node methods, and the LLM invocation logic. The graph flows through the following stages:

A typical run proceeds as follows. The agent begins in the DISCOVER phase, where it aligns column names across the input datasets (schema matching) and profiles each dataset to understand row counts, null rates, and column types. These profiles give the downstream LLM the context it needs to generate sensible pipeline code; for example, knowing that a “revenue” column is 40% null informs the choice of fusion resolver. The agent then enters the TRANSFORM phase, which applies normalization to source datasets (standardizing formats like country codes or date strings), runs the blocking tester to determine the most efficient candidate-pair generation strategy for each dataset pair, and runs the matching tester to find optimal comparator configurations. These three nodes produce frozen configurations that the pipeline generation node will translate into executable code. In the BUILD phase, the pipeline generation node assembles a comprehensive system prompt (including example code, dataset profiles, blocking/matching configurations, and any feedback from prior iterations) and asks the LLM to produce a complete Python integration script. The EXECUTE → ASSESS → DIAGNOSE loop then runs the generated pipeline, evaluates its output against a validation set, and and, if accuracy is below the quality gate, invokes the investigation agent to diagnose the root cause and route to the appropriate fix. This loop may iterate up to four times, with each iteration targeting a different pipeline stage based on the investigator’s evidence-driven diagnosis. Finally, the FINISH phase generates human-review tables with source lineage, runs the sealed final test evaluation against a held-out test set, and saves all results and audit artifacts.

Why Phases?

The graph is organized into five logical phases, each with a distinct responsibility. The DISCOVER phase (schema matching and data profiling) establishes a shared vocabulary and statistical understanding of the input data. Without this foundation, the LLM would have to infer column semantics and data characteristics from raw file previews alone, leading to frequent misalignment errors. The TRANSFORM phase (normalization, blocking tester, matching tester) prepares the data and determines empirically validated strategies. By testing multiple blocking and matching configurations against labelled test sets before pipeline generation, the agent removes guesswork from the LLM’s task: it only needs to translate a known-good configuration into code, not also invent the strategy. The BUILD phase (pipeline generation with optional self-review) produces the executable integration script. The EXECUTE → ASSESS → DIAGNOSE loop is the iterative refinement core: it runs the pipeline, measures quality, and uses the investigation agent to identify and fix the weakest link. The FINISH phase ensures that the final output is auditable and that evaluation is performed on a held-out test set to prevent overfitting to the validation set.

Conditional Routing Edges

Unlike Agents I-VI where the feedback loop always returns to pipeline generation, Agent VII has five distinct routing targets after the investigator node. The investigator LLM decides which stage to fix based on evidence:

Each conditional edge in the graph encodes a specific decision point. After pipeline_adaption (the code generation node), the should_continue_research() function checks whether the LLM has issued tool calls that still need to be resolved (for example, a documentation search query whose results have not yet been incorporated). If tool calls are pending, the node loops back to itself (up to 6 iterations) to process them; otherwise, it transitions to execute_pipeline. After execute_pipeline, the routing depends on the exit code: a successful run (exit code 0 with output files present) transitions to evaluation_node; a retryable failure (e.g., syntax error or import error) routes back to pipeline_adaption with the error message; exhausting all retry attempts terminates the run. After evaluation_node, the routing checks whether valid metrics were produced: if so, the flow continues to the investigator_node; if the evaluation script failed with a retryable error, it loops back for another evaluation attempt; if all retries are exhausted, the run ends. After the investigator_node, the LLM’s investigator_decision field determines the target: it may route to normalization_node (for format mismatches), run_blocking_tester (for candidate generation issues), run_matching_tester (for matching threshold or comparator problems), pipeline_adaption (for fusion strategy changes), or human_review_export (when quality is acceptable or attempts are exhausted). Finally, after human_review_export, the presence of a sealed test set determines whether the flow continues to sealed_final_test_evaluation or proceeds directly to save_results.

State Object

The SimpleModelAgentState TypedDict contains ~60 fields. The state object is the backbone of cross-node communication: because every node reads from and writes to the same shared state dictionary, information flows seamlessly between pipeline stages without explicit parameter passing. For example, the blocking tester writes its optimal configuration to blocking_config, which the pipeline generation node reads when assembling its system prompt; the evaluation node writes metrics to evaluation_metrics, which the investigator reads when diagnosing problems; and the investigator writes its decision to investigator_decision, which the conditional routing edge reads to determine the next node. The ~60 fields reflect the complexity of managing a multi-stage, iterative pipeline where each stage needs access to the outputs of all prior stages, the full history of what has been tried, and the current best results. Key additions beyond Agents I-VI include:

  • normalized_datasets: paths to normalized source datasets
  • fusion_guidance: per-attribute resolver recommendations with confidence scores and trust orderings from probes and investigation
  • cluster_analysis_result: per-pair cluster health, strategy ratings for all 5 algorithms, evidence summaries, sample problematic clusters
  • evaluation_cycle_audit: per-cycle records for building iteration history on-the-fly
  • iteration_history: list of per-attempt records including attempt, accuracy, delta, decision, and attribute_accuracies (per-attribute accuracy for cross-attempt learning)
  • pipeline_scaffold: frozen/mutable code sections after cycle 1
  • investigator_decision: routing string from the investigation LLM
  • auto_diagnostics: mismatch reasons, ID alignment, size comparison

Normalization Node

Agent VII introduces a dedicated normalization node that did not exist in Agents I-VI. The normalization node uses LLM-generated NormalizationSpec dicts via PyDI’s transform_dataframe(), ensuring stable, library-backed transformations rather than ad-hoc custom code.

Why Normalization Matters

Data integration frequently fails not because of wrong entity matches, but because of representation mismatches between sources. Consider the “country” attribute: one source might store "US" (ISO alpha-2), another "United States" (full name), and a third "USA" (ISO alpha-3). If the fusion engine picks the value from one source and the validation set expects another format, the evaluation reports 0% accuracy for that attribute, even though the underlying data is correct. Similarly, a revenue column might contain "1.2B" in one source and "1200000000" in another; date columns might use "2024-03-15" vs "March 15, 2024" vs "15/03/2024". Without normalization, these format differences propagate through blocking (reducing pair completeness because string comparisons fail on differently formatted values), matching (reducing similarity scores), and fusion (producing mixed-format output that evaluation penalizes).

The normalization node addresses this by standardizing source data formats before blocking and matching, so that downstream components operate on consistently formatted values. Critically, the LLM decides what to normalize by examining actual data samples from both the source datasets and the validation/target set, so the transformations are tailored to each specific integration task rather than applied generically.

How It Works

  1. Probe Source Data: the orchestrator (normalization_orchestrator.py, ~700 lines) samples 15 representative rows from each source dataset via build_dataset_probe(). It also samples rows from the validation/target set.
  2. LLM Spec Generation: the LLM receives the source data probes alongside the target set sample, enabling it to compare how values are formatted in each source versus what the target expects. For example, if the source probe shows country values like "United States" and "Germany" while the target sample shows "US" and "DE", the LLM can infer that country_format: "alpha_2" is the correct transformation. The LLM generates a NormalizationSpec per source dataset, choosing from the full set of available ColumnSpec fields. Available column transformations include:
    • output_type: string, float, int, bool, datetime, keep
    • case: lower, upper, title, keep
    • country_format: alpha_2, alpha_3, numeric, name
    • date_format: strftime patterns
    • expand_scale_modifiers: converts “1.2B” to 1200000000
    • stdnum_format: standardizes ISBNs, ISSNs, etc.
    • strip_whitespace, normalize_email, phone_format, etc.
  3. Apply Specs: _apply_spec_to_dataframe() validates columns, sanitizes ColumnSpec fields, preserves ID columns, and calls PyDI’s transform_dataframe(df, spec). Specs are saved to normalization_specs.json per attempt for reproducibility.

Acceptance Gating

Normalization is not blindly applied. Because normalization transforms the source data itself, a bad normalization can reduce accuracy; for example, if the LLM incorrectly lowercases a column where the validation set expects title case, or if it applies a date format that truncates time information that was previously correct. To guard against this, Agent VII implements an acceptance gate (investigator_acceptance.py, ~190 lines) that evaluates each normalization attempt against the previous best accuracy:

  • If normalization helped (accuracy increased), the normalized datasets are kept and become the new baseline for subsequent iterations.
  • If normalization hurt (accuracy decreased), datasets are rolled back to the pre-normalization versions, ensuring that the agent never permanently degrades data quality.
  • A consecutive rejection streak (default: 2) blocks further normalization attempts entirely. This prevents the agent from wasting investigation cycles on a normalization strategy that is not yielding results. After two consecutive rejections, the safety override forces the investigator to route to pipeline_adaption instead, so the agent shifts focus to fusion strategy improvements.
Why PyDI NormalizationSpec?

Earlier pipeline iterations wrote custom normalization code (ad-hoc Python transforms), which was flexible but unstable: a single LLM mistake could crash the pipeline. Agent VII uses PyDI’s declarative NormalizationSpec instead: each column gets a JSON spec with well-defined fields, and PyDI handles the actual transformation. The on_failure: "keep" option ensures that malformed values are preserved rather than crashing.

Investigation System

The investigation system is the intelligence core of Agent VII. It replaces the single-shot Reasoning Node of Agents III-VI with a multi-turn LLM-driven investigation agent that can analyze evidence, write diagnostic code, execute it, and make an informed routing decision.

To understand why this matters, consider a concrete scenario. After the first pipeline execution on the music use case, the evaluation reports 52% overall accuracy. The “label” attribute has only 45% accuracy, while “genre” is at 38%. In Agents III-VI, the Reasoning Node would see these aggregate numbers and might suggest “try a different fusion strategy for label,” but it cannot verify why label is wrong. Is it picking the wrong source? Is the format different? Are the entity matches themselves incorrect? The investigation system in Agent VII answers these questions empirically. The mismatch_sampler probe shows concrete examples: for record 42, the expected label is “Columbia Records” but the fused value is “Columbia,” classified as a value_mismatch. The source_attribution probe traces this back: MusicBrainz stores the full label name while Discogs stores the abbreviated version, and the current voting resolver is picking whichever appears most often rather than the most complete value. The investigator writes diagnostic code to confirm this pattern across all label mismatches, then routes to pipeline_adaption with a specific recommendation: “use longest_string for label.” The code guardrails then mechanically enforce this recommendation in the regenerated pipeline.

Orchestrator Flow

The investigation orchestrator (investigator_orchestrator.py, ~600 lines) manages the full investigation cycle. When overall_accuracy < QUALITY_GATE_THRESHOLD (0.85) and the attempt budget has not been exhausted (max 4 attempts), the orchestrator executes a carefully sequenced series of steps. Each step builds on the outputs of the previous ones, progressively assembling a complete evidence package before handing control to the investigation LLM:

  1. Evaluation Decision: reads the metrics produced by the evaluation script, runs sanity checks (are metrics in valid ranges? are any suspiciously perfect?), applies the regression guard to determine whether the new metrics represent genuine improvement over the previous best, and tracks the best-ever metrics across all attempts. This step is pure bookkeeping; it does not decide what to fix, but it establishes the factual foundation (what improved, what regressed, by how much) that every subsequent step relies on.
  2. Acceptance Feedback: evaluates whether the last normalization attempt (if any) actually helped. If the previous investigation cycle routed to normalization_node and the resulting accuracy is lower than before, the datasets are rolled back to their pre-normalization versions and the rejection streak counter is incremented. This prevents normalization damage from accumulating across iterations.
  3. Learning State: loads cross-run EMA (exponential moving average) gain tracking, which records which routing decisions have historically led to accuracy improvement for similar datasets. This provides the investigation LLM with additional context (“routing to normalization has historically yielded +3.2% gain for this dataset signature”) without making the decision for it.
  4. Run Probes: the 11 built-in probes execute and produce structured evidence (see Probe System). This is the most computationally significant step: probes read the fused output, the debug JSONL logs, the correspondence files, and the validation set to produce concrete, actionable summaries of what went wrong and where.
  5. Early Exit Check: if the quality gate is already met (which can happen after a normalization or matching tester improvement) or if the attempt budget is exhausted, the orchestrator routes directly to human review without invoking the investigation LLM, saving tokens and time.
  6. Cluster Analysis: if accuracy is still below 0.85, the orchestrator runs the cluster tester on the correspondence files to analyze cluster health and rate post-clustering algorithms (see Cluster Analysis). The cluster analysis results are added to the evidence package.
  7. Build Fusion Guidance: extracts mismatch classifications from the probes (e.g., “72% of label mismatches are value disagreements between sources”) and post-clustering recommendations from the cluster analysis, assembling them into a structured fusion_guidance object. This is a pure data transformation step with no LLM calls.
  8. LLM Investigation Loop: the core multi-turn loop (see below) where the investigation LLM receives all accumulated evidence and makes its diagnosis and routing decision.
  9. Enrich Fusion Guidance: after the investigation LLM has made its decision, this step bridges the gap between the LLM’s textual recommendations and the mechanical guardrails. It parses the LLM’s recommendation text to extract specific resolver names and trust map orderings per attribute, then populates attribute_strategies in the fusion guidance. These strategies are what the code guardrails read when deciding whether to override the next pipeline’s fusion calls.
  10. Safety Overrides: only 3 hardcoded overrides apply after the LLM has made its decision:
    1. Structural invalid correspondences (missing or malformed files) → force pipeline_adaption, because no other node can fix a broken correspondence file
    2. Normalization blocked (rejection streak reached or max normalization attempts exhausted) → force pipeline_adaption, preventing the agent from routing to a node that will be immediately rejected
    3. Invalid routing target (the LLM output an unrecognized node name) → default to pipeline_adaption as the safest fallback
  11. Save Transcript: the full multi-turn log, including every turn’s hypothesis, diagnostic code, execution output, and the final decision with its reasoning, is saved to investigation/investigation_<attempt>_<timestamp>.json. An iteration history entry (with per-attribute accuracies) is also appended to investigator_history.jsonl for cross-run learning.

The Multi-Turn Investigation Loop

The investigate() method in pipeline_agent.py implements a multi-turn conversation (max 4 turns, configurable via INVESTIGATION_MAX_TURNS). The multi-turn design is critical because many root causes cannot be diagnosed from the pre-computed evidence alone; the investigator often needs to inspect the actual data to confirm a hypothesis.

Consider a concrete scenario where multi-turn investigation is essential. On the music use case, the probes report that “genre” has 38% accuracy with mostly list_format_mismatch classifications. In Turn 1, the investigator hypothesizes that the genre lists use different separators across sources. It writes diagnostic code that loads the fused output and samples 10 genre values, printing them alongside the expected values from the validation set. The code runs in a subprocess and the output reveals that the fused genres use "; " as a separator while the validation set uses ", ". In Turn 2, the investigator now knows the root cause is a normalization issue (not a fusion issue), but it wants to check whether this pattern also affects other list columns like “styles.” It writes another diagnostic snippet, confirms that styles has the same problem, and then issues its decide action: route to normalization_node with the recommendation to standardize list separators. Without the multi-turn capability, the investigator would have had to guess from the aggregate list_format_mismatch classification, potentially misattributing the problem to the fusion resolver rather than the separator format.

On each turn, the LLM responds with a JSON object:

  • {"action": "investigate", "hypothesis": "...", "code": "..."}: the LLM writes diagnostic Python which is executed in a sandboxed subprocess (timeout 120s, stdin blocked). The stdout/stderr is captured and fed back as the next conversation turn. The LLM can use pandas, PyDI, and any standard library in its diagnostic scripts.
  • {"action": "decide", "next_node": "...", "diagnosis": "...", "recommendations": [...]}: the LLM makes its final routing decision. Available targets: normalization_node, run_blocking_tester, run_matching_tester, pipeline_adaption.
Why Multi-Turn?

In Agents III-VI, the Reasoning Node received metrics and debug events but could only passively reason about what went wrong. Agent VII’s investigator can actively test hypotheses, for example loading the fused output and checking whether a specific attribute’s values are misformatted, or examining the correspondence files to verify 1:many patterns. This transforms the investigation from speculation into empirical analysis. The multi-turn design also enables progressive refinement: the investigator might start with a broad hypothesis (“label accuracy is low because of source disagreement”), narrow it with code (“Discogs labels match 78% of the time vs MusicBrainz at 45%”), and then confirm the fix (“using longest_string would resolve 62% of the current mismatches”), all within a single investigation cycle.

Investigation Prompt

The investigation prompt (investigation_prompt.py, ~420 lines) teaches the LLM:

  • Its capabilities (analyze evidence, write/execute code, route to 4 targets)
  • What each routing target fixes (normalization for format issues, pipeline_adaption for fusion strategy, blocking/matching testers for upstream issues)
  • A Resolver Selection Guide with proven patterns (e.g., shortest_string for names, longest_string for labels, maximum for numeric fields, union for lists)
  • A Data Layout Guide documenting _fusion_sources, _fusion_metadata column formats so the LLM doesn’t waste investigation turns on data format discovery
  • Critical rules: “PROTECT WORKING ATTRIBUTES” and never route to normalization for value disagreements (only for format mismatches)

Bridging Recommendations to Guardrails

A central challenge in any LLM-driven system is ensuring that the LLM’s recommendations are actually implemented in the generated code. An LLM might correctly diagnose that “label should use longest_string” but then generate pipeline code that still uses voting, either because it forgot its own recommendation or because the pipeline prompt did not emphasize it strongly enough. Agent VII solves this with a two-layer mechanism that converts textual recommendations into mechanically enforced code patches.

After investigation, _enrich_fusion_guidance_with_strategies() converts the LLM’s textual recommendations into structured attribute_strategies that the code guardrails can mechanically enforce. These strategies come from two sources, each with a different confidence level:

  • Source attribution probe (confidence=0.7): data-driven trust ordering from per-source match rates against the validation set. For example, if the probe found that Discogs labels match the validation set 78% of the time while MusicBrainz labels match only 45%, the probe generates a strategy recommending prefer_higher_trust with Discogs ranked higher. The 0.7 confidence means this recommendation will be applied only if the LLM’s generated code does not already use a reasonable resolver for that attribute.
  • Investigation recommendations (confidence=0.9): parsed resolver names and trust maps from the LLM’s recommendation text. The enrichment function uses pattern matching to extract resolver names (e.g., longest_string, prefer_higher_trust) and trust map orderings from the natural-language recommendations. At 0.9 confidence, these override both the probe’s suggestion and the LLM’s own generated code, ensuring that the investigator’s diagnosis is faithfully executed.

This two-layer approach bridges the gap between the investigation LLM’s natural-language reasoning and the deterministic code guardrails that enforce the recommendations. The confidence-based priority system ensures that high-confidence investigation recommendations always take effect, while lower-confidence probe suggestions serve as a safety net when the LLM generates suboptimal code without a clear recommendation to override it.

Probe System

Agent VII introduces a system of 11 diagnostic probes (investigator_probe_runner.py, ~1100 lines) that run before each investigation cycle. Probes are evidence generators: they produce structured data that the investigation LLM interprets. No probe makes routing decisions.

The probe system exists because of a fundamental design insight: it is far more efficient to pre-compute structured evidence and present it to the LLM than to have the LLM figure out what to examine on its own. Without probes, the investigation LLM would need to spend its limited turns (max 4) writing exploratory code just to discover basic facts (which attributes are worst, what kind of mismatches dominate, which source is most trustworthy per attribute). By pre-computing this evidence deterministically, the probes free the LLM’s investigation turns for higher-level reasoning: confirming hypotheses, testing edge cases, and making nuanced routing decisions. In practice, this means the investigator often reaches a high-confidence diagnosis in Turn 1 and can make its routing decision immediately, reserving the multi-turn capability for genuinely ambiguous cases that require empirical verification.

Each probe is a deterministic Python function that reads from the agent state (fused output, debug JSONL logs, correspondence files, validation set) and returns a standardized dict with four fields: name (identifier), summary (human-readable findings), normalization_pressure (a float indicating how strongly the evidence points to normalization as the fix), and actionability_pressure (a float indicating how strongly the evidence suggests any action at all). The pressures are informational; they help the LLM prioritize but do not determine routing.

Highest-Signal Probes

Two probes consistently provide the most actionable evidence for the investigation LLM:

The mismatch_sampler probe is the single highest-signal evidence source. It reads the debug JSONL produced by the evaluation script, which contains per-record expected-vs-fused value pairs, and classifies each mismatch into a specific category. A case_mismatch (e.g., “columbia records” vs “Columbia Records”) points directly to normalization as the fix. A value_mismatch (e.g., “Columbia” vs “Columbia Records”) indicates that the fusion resolver is picking the wrong source. A missing_fused (the fused value is null or empty) suggests that the fusion strategy is dropping data. By presenting these concrete examples with their classifications, the probe gives the investigation LLM the specificity it needs to make targeted recommendations rather than generic suggestions.

The source_attribution probe is the most sophisticated evidence generator. For each low-accuracy attribute, it performs a multi-step analysis: first, it reads the fused output’s _fusion_metadata column, which contains Python repr-format dicts recording which source contributed each fused value; second, it parses these metadata entries (handling JSON, ast.literal_eval, and regex fallback for numpy types) to determine the source dataset for each fused value; third, it compares the fused value against the validation set using both exact and fuzzy matching; and finally, it computes per-source match rates. The result is a concrete recommendation: for example, “For attribute ‘label’: discogs matches 78% (exact: 65%, fuzzy: 78%), musicbrainz matches 45% (exact: 31%, fuzzy: 45%). Recommendation: longest_string or prefer_higher_trust(trust_map={"discogs": 3, "musicbrainz": 1}).” This probe also applies attribute-type-aware defaults: it recommends shortest_string for name-like attributes, longest_string for labels and artist names, maximum for numeric fields, and union for list-valued attributes.

Source Attribution Probe

The source_attribution probe is particularly powerful. For each low-accuracy attribute, it traces fused records back to their source datasets via _fusion_metadata (handling Python repr format with regex fallback for numpy types), computes exact and fuzzy match rates per source, and recommends the best resolver with trust ordering. For example, it might find that for the “label” attribute, Discogs has 78% match rate vs MusicBrainz at 45%, recommending prefer_higher_trust(trust_map={"discogs": 3, "musicbrainz": 1}). These recommendations feed directly into fusion_guidance["attribute_strategies"] for guardrail enforcement.

Cluster Analysis

The cluster analysis node was completely reworked for Agent VII. In Agents V-VI, the cluster tester could only recommend “AdjustMatchingConfig” or “None” and never suggested any of PyDI’s 5 post-clustering algorithms. Agent VII’s cluster tester (cluster_tester.py, ~650 lines) rates all algorithms with evidence.

Why Cluster Analysis Matters

Entity matching produces pairwise correspondences: “record A from Dataset 1 matches record B from Dataset 2 with score 0.87.” When integrating three or more datasets, these pairwise correspondences form clusters through transitivity. If A matches B and B matches C, then A, B, and C are all considered the same entity. This is correct when A and C genuinely refer to the same real-world object, but it becomes problematic when the transitive link is spurious. For example, in the games use case, a game titled “FIFA 23” in DBpedia might match “FIFA 23” in Metacritic (correct), and a different Metacritic entry for “FIFA 22” might match “FIFA 22” in the Sales dataset (correct), but if the matcher also produces a weak match between “FIFA 23” and “FIFA 22” in Metacritic (incorrect, but above threshold), transitive closure merges all four records into a single cluster, mixing data from two different games. Without post-clustering, these spurious transitive links propagate through the entire fusion output, producing records that blend attributes from unrelated entities.

Post-clustering algorithms address this by refining the raw correspondence graph. A MaximumBipartiteMatching algorithm enforces strict 1:1 assignments, keeping only the highest-scoring match for each entity and breaking spurious transitive chains. The choice of algorithm depends on the data characteristics: how many 1:many correspondences exist, how ambiguous the match scores are, and whether entities genuinely have multiple representations across sources. The cluster tester provides the evidence needed to make this choice.

Metrics Computed

For each dataset pair’s correspondence file, the tester computes three categories of metrics:

1. Cluster Health

  • Cluster size distribution (counts, percentages)
  • Small-cluster ratio (fraction of clusters of size 2)
  • Large-cluster ratio (fraction above threshold)
  • Long-tail detection (whether size distribution is heavily skewed)

2. Score Metrics

  • Mean, std, min, max, median
  • Percentiles: p25, p75
  • IQR (interquartile range), which indicates score spread
  • Low confidence ratio: fraction of pairs with scores below a threshold

3. Ambiguity Metrics

  • Per-side one-to-many ratio and max/mean fanout
  • Entities with multiple matches
  • Ambiguity ratio: fraction of entities where the top-2 match scores are within 0.1 of each other (hard to distinguish best match)
  • Count of ambiguous entities

Strategy Rating

Based on the computed evidence, rate_strategies() rates all 5 PyDI post-clustering algorithms plus “None” as recommended, viable, or unsuitable:

AlgorithmRecommended WhenTrade-off

MaximumBipartiteMatching One-to-many > 0.10 Optimal 1:1, but computationally expensive for large clusters
StableMatching High ambiguity (>0.3) with one-to-many > 0.10 Stable 1:1 assignments (Gale-Shapley); good when scores are close
GreedyOneToOneMatchingAlgorithm IQR > 0.15 and one-to-many > 0.05 Fast greedy highest-score-first; good when score separation is clear

Sample Clusters

The tester also extracts sample problematic clusters for the investigation LLM to inspect: the largest connected components (with scores) and the most ambiguous entities (sorted by smallest score gap between top-2 matches). These concrete examples help the LLM decide whether post-clustering is needed and which algorithm suits the data pattern.

Key Design Decision

The cluster tester is purely deterministic and does not call an LLM (unlike the main-branch version which accepted an llm parameter that was never used). All strategy ratings are evidence-based and advisory: the investigation LLM makes the final decision about whether to apply post-clustering and which algorithm to use.

Pipeline Self-Review

Agent VII introduces a two-stage self-review that runs inside the pipeline generation node, between code generation and execution. This catches structural issues, overcorrection risks, and inconsistencies before the pipeline is executed, saving expensive subprocess runs on code that would fail or produce poor results.

How It Works

The self-review is triggered only when static sanity checks find actual issues in the generated code. If no issues are found, the review is skipped to save tokens.

Stage 1: Critique

The generated pipeline code is sent to the LLM along with a structured evidence payload containing:

  • Problem classes from the evaluation decision
  • Pipeline execution results and evaluation metrics
  • Auto diagnostics and regression guard status
  • Investigator action plan and fusion guidance
  • Static sanity findings: deterministic checks that catch common issues (e.g., missing correspondence files, invalid API usage, inconsistent column names)

The review LLM classifies issues into five categories: structural, identity/provenance, representation/serialization, value-selection/fusion, and overcorrection risk. It returns a JSON verdict: 

{
  "verdict": "pass" | "revise",
  "summary": "short explanation",
  "problem_classes": ["..."],
  "keep": ["things that are correct"],
  "risks": ["potential issues"],
  "revision_instructions": ["targeted fixes"]
}

Stage 2: Targeted Revision

If the verdict is "revise", a second LLM call receives the current pipeline code and the review’s revision instructions. The revision prompt enforces strict constraints:

  • Minimal changes only: preserve working sections
  • Preserve trust_map and dataset-identifier literals unless the review explicitly identifies them as wrong
  • Prefer PyDI built-in fusers over custom helpers
  • Keep the fusion layer PyDI-native: avoid introducing new helper functions
  • If the review is ambiguous, leave the code unchanged
Why Separate Critique from Revision?

In Agents I-VI, LLM-generated code went directly to execution. If it had issues, the agent only discovered them after a failed run or poor evaluation , wasting an entire execution+evaluation cycle. The self-review catches issues proactively, but its two-stage design (critique then revise) is equally important. When a single LLM call is asked to both identify problems and fix them, it tends to overcompensate, rewriting large sections of working code, changing variable names, or introducing new bugs while fixing old ones. By separating the roles, the critic operates with a purely analytical mindset: it identifies specific, categorized problems and marks what is working correctly (the "keep" list). The reviser then receives explicit, bounded instructions and is constrained to make minimal changes. This division of labor produces more reliable fixes and preserves the integrity of working pipeline sections.

Code Guardrails

One of the most impactful innovations in Agent VII is the comprehensive code guardrails system (code_guardrails.py, ~1200 lines). In Agents I-VI, LLM-generated code was executed as-is, meaning common LLM mistakes (invalid API arguments, missing imports, wrong output paths) would crash the pipeline. Agent VII applies mechanical fixes to every generated script before execution.

The philosophy behind mechanical guardrails is that certain categories of errors are predictable and fixable without LLM intervention. Prompt instructions asking the LLM to “always use valid list_strategy values” or “remember to import all resolvers” are inherently unreliable: the LLM may comply 90% of the time, but the remaining 10% wastes an execution cycle on a crash that could have been prevented with a simple regex-based patch. By encoding these fixes as deterministic code transformations, the guardrails achieve 100% reliability for known error categories, freeing the LLM to focus on the genuinely creative aspects of pipeline generation (choosing fusion strategies, setting trust orderings, deciding post-clustering approaches) rather than getting bogged down in API compliance details. The guardrails also serve as the enforcement mechanism for the investigation system’s recommendations: when the investigator recommends a specific resolver for an attribute, the guardrails mechanically patch the generated code to use that resolver, regardless of whether the LLM remembered the recommendation.

Pipeline Guardrails (apply_pipeline_guardrails())

1. List Strategy Validation

Each PyDI comparator type has a different set of valid list_strategy values. The LLM frequently generates invalid combinations. The guardrails validate and fix:

ComparatorValid StrategiesCommon LLM Mistake
StringComparator best_match, set_jaccard, set_overlap "concatenate" → upgraded to "best_match"
NumericComparator average, best_match, range_overlap, set_jaccard Using string-only strategies
DateComparator closest_dates, range_overlap, average_dates, latest_dates, earliest_dates Using best_match (not valid for dates)

2. Fusion Guidance Enforcement

When the investigation system recommends specific resolvers for attributes, the guardrails mechanically patch add_attribute_fuser() calls in the generated pipeline:

  • High confidence (≥0.85): the guardrail overrides the LLM’s choice with the recommended resolver, even if the LLM picked something different.
  • Low confidence (<0.85): the guardrail respects the LLM’s built-in choice.
  • Per-attribute trust maps: when attributes need conflicting trust orderings (e.g., label wants discogs>musicbrainz but duration wants musicbrainz>discogs), the guardrails inject separate trust_map_<attr> variables per attribute.

3. Import Safety

The guardrails scan all add_attribute_fuser() calls for unimported PyDI resolver symbols and auto-add the missing imports. This prevents NameError crashes from the LLM forgetting to import a resolver.

4. Eval ID Injection

Detects the validation set ID prefix (e.g., mbrainz_ for the music use case) and injects eval_id column extraction into the pipeline code. This ensures the fused CSV always maps cluster IDs back to validation-aligned source IDs.

5. ML Post-Clustering Safety

When matcher_mode="ml" and no post-clustering exists in the generated code, the guardrails inject MaximumBipartiteMatching on each pairwise correspondence set before concatenation. This prevents RecursionError from large connected components in ML matcher output.

6. Other Fixes

  • Force include_singletons=True on all fusion calls
  • Rewrite hardcoded output/ paths to the run-scoped directory
  • Remove dangerous patterns (e.g., sys.exit(), os.remove())
  • Freeze matching thresholds unless the investigator explicitly routed to run_matching_tester

Evaluation Guardrails (apply_evaluation_guardrails())

  • Sub-column stripping: drops fused sub-columns (e.g., tracks_track_name) when the gold set only has the parent column (tracks), preventing 0% accuracy entries that waste investigation cycles.
  • Numeric type coercion: casts year/count/score/page columns to Int64 in both fused and gold DataFrames, preventing 274.0 ≠ 274 false mismatches.
  • List separator normalization: aligns list separators between fused and gold (e.g., "; " → ", ") for genre/category/tag columns.
  • Output path rewriting: redirects hardcoded paths to the run-scoped directory.

Pipeline Scaffold

In Agents I-VI, every iteration regenerated the entire pipeline from scratch, including blocking, matching, and fusion code. This was wasteful because the blocking and matching strategies are determined by the tester nodes and should not change between iterations. Agent VII introduces pipeline_scaffold.py (~315 lines), which splits the pipeline into frozen and mutable sections.

How It Works

  1. After cycle 1: the scaffold extractor identifies code landmarks using regex patterns:
    • Frozen sections: imports, dataset loading, blocking, matching, correspondence saving, which are deterministic and proven to work.
    • Mutable sections: post-clustering strategy, fusion strategy/resolvers, which are what the investigator recommends changing.
  2. On subsequent cycles: the LLM only generates the mutable sections. The scaffold splices the new code back into the frozen infrastructure using regex landmark detection.

Concrete Example

Consider the music use case after a successful first execution. The pipeline code is approximately 250 lines long. The frozen sections include: the import block (~20 lines), dataset loading and ID column setup (~15 lines), three blocking configurations with TokenBlocker calls (~30 lines), three matching configurations with RuleBasedMatcher calls and comparator definitions (~60 lines), and correspondence file saving (~15 lines), roughly 140 lines of code that was empirically validated by the blocking and matching testers and executed successfully. The mutable sections are: the post-clustering strategy (~20 lines choosing and applying a clustering algorithm to each correspondence set), the fusion strategy setup (~40 lines defining the DataFusionStrategy with per-attribute add_attribute_fuser() calls), and the fusion execution and output saving (~30 lines), roughly 90 lines. On the next iteration, when the investigator routes to pipeline_adaption with a recommendation to change the label resolver from voting to longest_string, the LLM only regenerates those 90 mutable lines. The scaffold splices the new fusion code back into the frozen blocking/matching infrastructure, producing a complete pipeline that preserves all working upstream logic while implementing the targeted fix.

Benefits

  • Reduced token usage: the LLM generates ~30% of the code instead of 100%, significantly reducing both cost and latency per iteration.
  • No accidental regressions: working blocking/matching code cannot be accidentally broken by the LLM during a fusion-focused iteration. This is particularly important because the blocking and matching configurations were determined by dedicated tester nodes with empirical validation, and regenerating them risks introducing subtle errors that the testers would not catch.
  • Focused iteration: the LLM concentrates on the sections that actually need improvement, producing higher-quality changes because its attention is not diluted across 250 lines of boilerplate.

Other Subsystems

Regression Guard

Agents I-VI had no regression protection: a pipeline that regressed from 60% to 20% accuracy was treated the same as one that improved. Agent VII implements a sophisticated net-benefit weighted regression guard (metrics.py, ~260 lines) that prevents the agent from accepting a pipeline iteration that makes things worse overall, while still allowing iterations that improve most attributes even if one or two regress slightly:

  1. Compute gains and drops per attribute between the new and previous best metrics.
  2. Classify severity of each drop: minor (0.02), moderate (0.08), severe (0.12), catastrophic (0.20).
  3. Apply net-benefit calculation: if total gains > total losses, allow the update despite some drops.
  4. Reject only when: overall/macro regression ≥ 0.02 with no compensating gains, OR catastrophic drops that outweigh improvements.

To illustrate why net-benefit weighting matters, consider a concrete scenario from the music use case. The previous best pipeline achieved 68% overall accuracy with per-attribute scores: label 45%, genre 38%, duration 82%, country 71%, year 95%. The investigator recommends switching the label resolver from voting to longest_string. The new pipeline achieves: label 72% (+27%), genre 41% (+3%), duration 79% (−3%), country 71% (0%), year 95% (0%), for 72% overall. A naive regression guard would reject this update because duration dropped by 3%. The net-benefit guard correctly accepts it: the +27% gain on label and +3% on genre far outweigh the −3% minor drop on duration. However, if a different iteration produced label 75% (+3%) but duration 42% (−40%), the guard would reject it because the catastrophic 40% drop on duration is not compensated by a modest 3% gain on label.

Reasoning Case Study: Three Runs on the Games Dataset

The preceding sections describe what Agent VII is capable of. This section looks at how it actually reasons in practice, by observing three independent runs of the agent on the Games benchmark and inspecting the investigator's trace at each decision point. The goal is to move past headline accuracy numbers and ask whether the agent is genuinely reasoning about the pipeline or merely narrating around a randomised routing policy.

Because investigator_node writes a structured JSON record at every turn — containing a diagnosis, a reasoning paragraph, a routing decision, a structural-ceiling estimate, and per-attribute recommendations with impact estimates and improvability classes — the agent's thought process is directly inspectable. All quotes in this section are verbatim from the investigator JSON files at output/runs/<run_id>/investigation/investigation_*.json.

Experimental Setup

The agent was invoked three times in sequence on the Games dataset (dbpedia.xml, metacritic.xml, sales.xml), with the blocking and matching testers pre-loaded with validated configurations so that only the pipeline-adaption / evaluation / investigator loop was exercised. Datasets, test sets, matcher mode (rule-based), recursion limit, and the underlying LLM (gpt-5.4) were held constant across all three runs. Each run produced its own isolated output/runs/<run_id>/ directory with a complete decision trace.

Run 1 — The Two-Stage Repair (48.75% → 75% → 85%)

The first attempt produced 48.75% overall accuracy. The investigator's diagnosis has the structure of a well-written root-cause analysis: it first rules out two hypotheses, then names the actual culprit, then produces evidence anchored in the fused output itself with specific mismatch examples, then brings in quantitative cluster statistics to corroborate the qualitative examples:

"The dominant error is not normalization or fusion resolver choice; it is over-linking/ambiguous entity matching that merges different games into the same cluster. Evidence: worst mismatches are semantic substitutions across title/version/platform/year (e.g. Plants vs Zombies: Garden Warfare 2 fused as Lego Star Wars: The Force Awakens; NBA 2K12 fused as Madden NFL 07; Civilization V fused as Civilization IV). Those are wrong entity pairs, not formatting differences. Cluster diagnostics reinforce this: one-to-many ratios of 62-66%, ambiguity ratio >71%, and extremely large clusters (max_size >1000) indicate many competing candidate links and insufficient disambiguation. The current fusion then propagates wrong matched records, causing catastrophic releaseYear/platform/name/developer errors. Normalization already aligns simple formats, and the worst attributes are explicitly assessed as structurally limited at current matching structure. Fusion changes alone cannot repair clusters that already contain the wrong records."

The investigator then routes to run_matching_tester, explicitly framing the decision as a risk-minimisation problem over the attributes that are already above 85%:

"Matching should be fixed next because blocking/matching artifacts show high ambiguity despite decent pairwise F1, and the observed output errors are classic false-positive matches between related but distinct games/franchises/editions/ platform releases. The protected attributes above 85% should be preserved; changing fusion globally risks regressing them while leaving the underlying wrong-entity clusters untouched."

On the second attempt, with overall accuracy now at 75%, the investigator does something that distinguishes Run 1 from the other two: it reads the generated fusion code and identifies a genuine bug. The maximum resolver, which is numeric, had been applied to the string-valued name column, silently producing null outputs:

"The primary actionable defect is in fusion logic, specifically the name attribute being fused with the wrong resolver. The pipeline currently uses maximum for name, which is a numeric-oriented resolver and is producing an entirely null name column (100% null, 0% accuracy) despite all three sources having 100% non-null names and metacritic matching the validation set exactly 65% of the time. This explains the catastrophic regression of name_accuracy from 40% to 0%."

This is no longer pattern-matching on metrics; it is a specific causal chain. The investigator then routes to pipeline_adaption and proposes a concrete textual replacement, with trust-map weights derived from the source-vs-validation exact-match oracle:

"Replace strategy.add_attribute_fuser("name", maximum) with strategy.add_attribute_fuser("name", prefer_higher_trust, trust_map={"metacritic": 3, "sales": 2, "dbpedia": 1})."

The third attempt reaches 85%, the quality gate fires, and Run 1 exits through human review, landing at 82.4% on the sealed test.

Run 2 — The Conservative Loop (42.5% → 80% → 85%)

Run 2 is the counterfactual. It starts from almost exactly the same position as Run 1 — similar overall accuracy, the same mismatch patterns, the same cluster statistics — but the investigator never leaves the matching branch. Both of its turns route to run_matching_tester, and the quality gate is reached not by surgical intervention but by the matching rerun happening to produce cleaner correspondences on the second try.

What is remarkable is that the investigator's first-attempt reasoning does explicitly acknowledge the post-clustering recommendation that Run 3 will later act on:

"Post-clustering recommendation of MaximumBipartiteMatching supports the same conclusion, but the core issue still originates in overly permissive/ambiguous matches; the matching stage must be tightened and re-evaluated."

The investigator sees the suggestion from the cluster-evaluation subnode and mentions it by name, but interprets it as a downstream symptom rather than an actionable fix. The claim is that the root cause is upstream in matching, so patching the post-clustering step would be treating the symptom. This is a defensible reading of the evidence, but it defers the cheapest available fix.

An incidental but important observation from Run 2: between attempts 1 and 2 the investigator's structural-ceiling estimate rose from 0.575 to 0.8625. The ceiling is not a fixed property of the data but a belief state that the agent updates as more evidence arrives. That self-correcting behaviour is arguably more interesting than the estimates themselves.

Run 3 — The Surgical Fix (52.5% → 90%)

Run 3 is the most efficient of the three, converging in two iterations rather than three. Its first-attempt diagnosis is structurally identical to Runs 1 and 2 — same root cause, same evidence, same quantitative framing — but the final clause of the reasoning paragraph makes a different routing choice, and explicitly argues for it over the alternatives:

"Because blocking and pairwise matching outputs are already reasonably dense and non-empty, but downstream clusters are many-to-many and highly ambiguous, the most surgical next fix is in frozen-stage-allowed fusion/post-clustering logic: add the recommended MaximumBipartiteMatching post-clustering before fusion. This directly addresses the observed cross-game contamination while preserving the 14 attributes already above 85%. I am explicitly not recommending normalization first because the evidence is not format-driven overall; and not routing to run_matching_tester because the provided cluster analysis already identifies the actionable remedy within pipeline adaptation: post-clustering."

The phrase “I am explicitly not recommending … and not routing to run_matching_tester” is notable because the investigator is reasoning over alternatives rather than making a single best-match pick. It considers three candidate routes, gives an explicit reason for rejecting two of them, and lands on adaption. The fix moves validation accuracy from 52.5% to 90% in a single iteration, the quality gate fires, and Run 3 exits after just two investigator turns — using 37% fewer tokens than the other two runs.

Reasoning Habits the Investigator Exhibits Consistently

Across all three runs the investigator displays a set of reasoning habits that are not at all guaranteed from a raw LLM. These are the stronger positive signal of this experiment: the diagnosis is reproducible even when the routing is not.

  • Grounding in concrete mismatch examples. Every run quotes specific records from the fused output ("Plants vs Zombies: Garden Warfare 2 fused as Lego Star Wars", "NBA 2K12 fused as Madden NFL 07", "Zelda Twilight Princess HD fused as Zelda II"). The diagnosis is anchored in what actually went wrong, not in a generic restatement of the metrics.
  • Distinguishing entity errors from formatting errors. Phrasings like "semantic substitutions across title/version/platform/year" and "cross-entity value mismatches rather than systematic formatting differences" appear across all three runs. This distinction is necessary to avoid pointless normalization reruns, and every investigator turn makes it correctly.
  • Quantifying the evidence. Diagnoses cite one-to-many ratios, ambiguity ratios, cluster sizes, matcher F1 scores, and source-vs-validation exact-match rates. When Run 2 writes that "matching F1 around 0.86-0.89 looks decent globally, but cluster diagnostics reveal pairwise ambiguity severe enough to corrupt downstream entity fusion", it is resolving an apparent contradiction with a specific mechanism.
  • Protecting good attributes from collateral damage. Every run contains at least one sentence about not touching attributes above 85%. This is what makes the agent's iteration monotonic in practice rather than oscillating.
  • Classifying improvability. Each attribute recommendation is tagged improvable or structurally_limited, the latter justified with variants of "source-vs-validation exact rates are near zero". This prevents the investigator from prescribing fixes that cannot work — no fusion strategy can recover a value none of the sources holds.

Observed Weaknesses

  1. Structural-ceiling miscalibration. The first-attempt ceiling estimates were 0.575, 0.575, and 0.586. Actual achieved accuracies were 0.85, 0.85, and 0.90 — the estimate is off by 25-30 percentage points. Inspecting the reasoning shows why: the ceiling is a "do nothing" estimate that does not account for the investigator's own proposed fixes. It is useful as a relative trend signal (Run 2's ceiling rose from 0.575 to 0.8625 between attempts) but not as an absolute achievability cap.
  2. Path-dependent routing under identical evidence. Runs 1, 2, and 3 see the same cluster-analysis recommendation and reach different routing decisions. The choice is partly a function of LLM sampling variance, not purely of the evidence. In practice it is not catastrophic — all three runs converge on a reasonable outcome — but it means the routing policy is not reproducible even when the diagnosis is.

Implications for Agent VII

  1. Agent VII's investigator is genuinely reasoning about the pipeline, not narrating around a random policy. The three runs independently converge on the same root cause using the same evidence, the same attribute classifications, and the same protected set. Structural properties of the correspondences drive the diagnosis, not the top-line accuracy number.
  2. The routing policy is the weak link. All three runs had access to the same cluster-analysis recommendation (MaximumBipartiteMatching post-clustering), and only Run 3 acted on it within pipeline_adaption. A heuristic that weights cluster-analysis recommendations more heavily when they name a strategy applicable without rerunning matching would likely close this gap.
  3. The code-auditing capability is a genuine strength. Run 1 identified a specific bug in the generated fusion code (maximum resolver on a string column), explained the causal chain, and proposed a specific replacement with oracle-derived trust-map weights. That kind of in-the- loop code audit justifies keeping the investigator node even once the rest of the pipeline is stable.
  4. The quality gate fires on a noisy signal. At n=10 validation records a single flip moves overall accuracy by 10 percentage points. Run 3's 0.90 vs 0.85 validation "win" over the other two is sampling noise, not a capability gap — all three converge on ~83% sealed test. Interpreting validation improvements should take this into account.

In short: the investigator's reasoning is consistent, evidence-grounded, and risk-aware. Its routing is not fully deterministic, but the decisions it reaches are defensible variants of the same causal model. For a project assessment, this is arguably the most meaningful outcome — the agent is not just producing correct outputs, it is producing correct outputs for the right reasons, and those reasons are directly readable from the investigator JSON trace.
