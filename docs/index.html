Data Integration & LLMs

Using LLM-Agents in Data Integration Pipelines

Authors: Onur Can Memis, Gregor Debus, Abdullah Momin, Nicolas Blazek

Introduction

Welcome to the documentation for our team project. This site explains how we use LLM-based agents to build robust and scalable data integration pipelines.

Status

This project is currently under active development. Some features may still be experimental.

Project Overview

Data integration is the process of combining data from multiple sources into a unified view. Our project explores how modern Large Language Models (LLMs) can be integrated into traditional pipelines to handle complex tasks like schema matching, entity resolution, and data fusion.

Core Goals

  • Automation: Reduce manual intervention in mapping and merging data.
  • Accuracy: Leverage LLM reasoning to improve matching precision.
  • Scalability: Design a pipeline that handles diverse datasets from multiple domains.

Architecture

Our data integration system is built as a modular, agent-based pipeline using LangGraph state graphs. Each agent version extends the previous one with additional capabilities, enabling us to evaluate the incremental value of each component. All agents share a common core (schema matching, dataset profiling, LLM-driven pipeline generation, code execution, and evaluation) but differ in the upstream analysis stages and the feedback loops they support.

Legend

Shared State

Every agent variant operates on the same AgentState typed dictionary that is threaded through the LangGraph graph. Key fields include the list of input dataset paths, dataset profiles, the generated pipeline code, execution results and attempt counters, evaluation code and metrics, and for the more advanced agents blocking configurations, matching configurations, schema correspondences, and the matcher mode (rule_based or ml). Because the state is shared, each node can read the outputs of all prior nodes and every re-entry into the generation node carries the full history of errors and evaluation feedback.

Common Nodes

Several processing nodes appear across all or most agent variants. They form the backbone of the pipeline and are described here once to avoid repetition.

Match Schemas

The Match Schemas node aligns column names across the input datasets. It invokes an LLM-based schema matching routine that receives the raw column lists from every source and produces a dictionary of correspondences. These correspondences are persisted in the state and consumed by all downstream nodes so that blocking, matching, and fusion operate on a unified attribute namespace. If correspondences are already present in the state (e.g. from a previous invocation), the node is skipped.

Profile Datasets

The Profile Datasets node uses an LLM tool call to invoke the ProfileDatasetTool on every input dataset. The tool returns a JSON profile containing row counts, null percentages per column, and column data types. The profiles are saved to disk and stored in the agent state so that the pipeline generation node can reason about data characteristics such as sparsity and type mismatches.

Generate Pipeline

The Generate Pipeline node is the core LLM code-generation step. It assembles a large system prompt that includes an example integration pipeline using the PyDI library. It consumes the list of dataset paths, first-row previews, dataset profiles, etc. It also conditionally includes schema correspondences and evaluation reasoning from prior iterations. The LLM is asked to produce a complete Python script that performs blocking, entity matching, and data fusion.

On the first invocation the prompt simply says "Create the integration pipeline." On subsequent invocations it includes the previously generated code together with any execution errors or evaluation feedback, asking the LLM to fix or improve the script.

Execute Pipeline

The Execute Pipeline node runs the generated Python script as a subprocess. If the process exits with a non-zero return code, the stderr output is captured and fed back to the Generate Pipeline node. The agent retries up to a maximum number of attempts before terminating. A successful execution transitions the workflow to the evaluation branch.

Generate Evaluation Code & Execute Evaluation

After a successful pipeline run the agent generates evaluation code following a similar pattern: a system prompt that includes an example evaluation script, the generated pipeline code, the fused output path, and a preview of the gold-standard test set. The generated evaluation script loads the fused output and the test set, computes per-column and overall accuracy, and writes structured metrics to and output file. The evaluation code is then executed as a subprocess with the same retry logic as the pipeline execution step.

Evaluation Decision

The Evaluation Decision node reads the JSON metrics produced by the evaluation script. It checks two termination criteria: either the overall fusion accuracy exceeds a threshold (85%) or the maximum number of evaluation attempts (three) has been reached. If neither condition is met, the workflow continues to the next stage, which, depending on the agent variant, is either termination or the reasoning node.

Agent I: Basic Pipeline Creation (RB)

Agent I is the simplest variant and establishes the baseline. It supports only rule-based matching and does not include any upstream analysis for blocking or matching strategy selection. Its workflow is a straight-line sequence with two conditional retry loops.

Workflow

  1. Profile Datasets: characterise each input dataset.
  2. Generate Pipeline: the LLM receives the example pipeline, dataset profiles, and first-row previews, and produces a complete integration script. No blocking or matching configuration is provided; the LLM must infer suitable strategies from the data alone.
  3. Execute Pipeline: run the generated script. On failure, loop back to Generate Pipeline with the error message (max. 3 attempts).
  4. Generate Evaluation Code: create an evaluation script for the fused output.
  5. Execute Evaluation: run the evaluation. On failure, loop back to Generate Evaluation Code (max. 3 attempts).
  6. Evaluation Decision: if overall accuracy ≥ 85% or attempts ≥ 3, terminate.

Because Agent I gives the LLM no pre-computed strategy information, the quality of the generated pipeline depends entirely on how well the model can infer reasonable blocking keys and matching rules from the dataset profiles and the example code. This makes Agent I a useful lower bound for comparison.

Agent II: Blocking & Matching Tester (RB)

Agent II extends Agent I by inserting two dedicated evaluation nodes before pipeline generation: the Blocking Tester and the Matching Tester. Both nodes run automated experiments to determine the optimal strategy for each dataset pair and persist the results in the agent state. Like Agent I, Agent II supports only rule-based matching.

Blocking Tester

The Blocking Tester receives the schema-aligned datasets and the entity-matching test sets. For each dataset pair it evaluates multiple blocking strategies: Standard Blocker (exact equality), Token Blocker (shared tokens), Sorted Neighbourhood Blocker (sliding window), and Embedding Blocker (semantic similarity via sentence transformers). Each strategy is scored by its Pairs Completeness (PC), which measures the proportion of true matches retained after blocking. The best strategy per pair is selected subject to a PC threshold (default 0.9) and a maximum candidate count. The resulting blocking configuration is a JSON object specifying, for each pair, the blocker type, the blocking columns, and the relevant parameters (e.g. top_k for the Embedding Blocker or window for the Sorted Neighbourhood Blocker).

Matching Tester

The Matching Tester operates on the candidate pairs produced by the optimal blocking strategy. It uses the LLM to propose comparator configurations (StringComparator, NumericComparator, DateComparator) with associated weights and thresholds for a rule-based matcher. Multiple configurations are evaluated against the test sets and scored by F1. The best configuration per pair is retained in a matching configuration JSON that includes comparator types, column mappings, weights, preprocessing steps, and the decision threshold. The tester iterates up to a configurable number of attempts, feeding evaluation results back to the LLM to refine the configuration.

Workflow Extension

  1. Match Schemas
  2. Profile Datasets
  3. Blocking Tester: determine optimal blocking strategy per pair.
  4. Matching Tester: determine optimal matching strategy per pair.
  5. Generate Pipeline: now receives blocking and matching configurations as additional context in the system prompt.
  6. Execute → Evaluate → Decide (same as Agent I).

By providing pre-computed, empirically validated blocking and matching configurations, Agent II removes much of the guesswork from the pipeline generation step. The LLM can focus on correctly translating the configuration into executable code rather than also having to invent a strategy.

Agents III & IV: Reasoning (RB & ML)

Agents III and IV introduce a Reasoning Node that closes the feedback loop between evaluation and pipeline generation. After the Evaluation Decision determines that accuracy is below the threshold, instead of simply re-running the generation node with the raw metrics, the workflow first passes through the Reasoning Node which performs a structured analysis of why the pipeline underperformed.

Agent III uses a rule-based matcher, while Agent IV uses a machine-learning-based matcher. The ML variant changes the example pipeline, the matching tester configuration, and the prompts to use feature extraction, classifier training, and MLBasedMatcher instead of weighted comparators and RuleBasedMatcher.

Reasoning Node

The Reasoning Node receives the aggregate evaluation metrics, a sample of detailed per-record debugging events from the fusion evaluation (up to 50 JSONL records), the current pipeline code, the evaluation code, and the blocking and matching configurations. It is prompted to identify dominant error types (false positives, false negatives, missed blocks, bad matches) and to determine whether the errors originate in blocking, matching, schema alignment, preprocessing, or the fusion logic. The node produces concise, structured improvement suggestions that are injected into the next pipeline generation prompt as additional context.

Because the blocking and matching strategies have already been independently validated by the tester nodes, the Reasoning Node is instructed to focus its suggestions on the fusion stage e.g., choosing different attribute fusers, adjusting trust strategies, or fixing preprocessing inconsistencies that cause column mismatches during evaluation.

ML Matcher Mode (Agent IV)

When the matcher mode is set to ml, several components adapt automatically. The Matching Tester proposes comparator features for a FeatureExtractor rather than weights for a rule-based decision function. The generated pipeline trains a binary classifier on the labelled test set using the extracted features and applies it via MLBasedMatcher. The evaluation and reasoning nodes operate identically; only the matching strategy and example code differ.

Workflow Extension

  1. Match Schemas → Profile → Blocking Tester → Matching Tester (same as Agent II).
  2. Generate Pipeline → Execute Pipeline (with retry loop).
  3. Generate Evaluation → Execute Evaluation (with retry loop).
  4. Evaluation Decision: if accuracy < 95% and attempts < 3, continue to Reasoning Node; otherwise terminate.
  5. Reasoning Node: analyse errors, produce improvement suggestions.
  6. Loop back to Generate Pipeline: the suggestions are appended to the system prompt, enabling the LLM to make targeted corrections in the next iteration.

Agents V & VI: Cluster Analysis & Documentation Search Tool

Agents V and VI represent the most complete version of the system. They retain all capabilities of Agents III/IV and add two new components: a Cluster Analysis node and a Search Documentation Tool. Agent V uses rule-based matching; Agent VI uses ML-based matching.

Cluster Analysis

After the pipeline produces entity correspondences, the Cluster Analysis node examines the cluster distribution of the matched entities. It computes cluster sizes, identifies the frequency of each cluster size, and derives diagnostic ratios: the small-cluster ratio (fraction of clusters of size 2), the large-cluster ratio (fraction above a threshold), and the one-to-many ratio. A high one-to-many ratio or the presence of very large clusters indicates over-matching, which the node diagnoses and addresses by recommending a strategy e.g., AdjustMatchingConfig with a higher threshold.

The cluster analysis is fed into subsequent iterations of the pipeline generation so that the LLM can tighten matching thresholds or adjust blocking parameters to reduce spurious clusters. This proved especially valuable for the Books use case, where editions of the same title (e.g. multiple printings of The Fellowship of the Ring) created clusters of size 9–30.

Search Documentation Tool

The Search Documentation Tool gives the pipeline generation node access to the PyDI library documentation at generation time. When enabled, the agent issues targeted queries to retrieve documentation for specific classes and functions e.g., the parameters of EmbeddingBlocker, the available fuser functions in DataFusionEngine, or the correct usage of RuleBasedMatcher comparators. The retrieved documentation snippets are appended to the system prompt, reducing hallucinated API calls and improving first-attempt success rates.

Enabling the documentation tool increases token usage considerably (since documentation text is injected into the prompt), so it is controlled by a global flag (INCLUDE_DOCS) and used selectively.

Workflow Extension

  1. Match Schemas → Profile → Blocking Tester → Matching Tester.
  2. Generate Pipeline: now additionally augmented with documentation search results when INCLUDE_DOCS is enabled.
  3. Execute Pipeline.
  4. Cluster Analysis: diagnose the entity cluster distribution.
  5. Generate Evaluation → Execute Evaluation.
  6. Evaluation Decision → Reasoning Node (conditional loop).

Methodology

This section describes the experimental methodology that governs all pipeline runs reported in this project. It covers the software stack, the incremental design of the agent variants, the variables that are held constant across experiments, and the protocol used to measure pipeline quality.

Experimental Setup

All experiments are built on a common software stack. The orchestration layer uses LangGraph, a state-graph framework from the LangChain ecosystem that models each pipeline as a directed graph of typed nodes with conditional edges and retry loops. Every LLM call is routed to OpenAI GPT 5.4, which serves as the single generative model for schema analysis, code generation, evaluation authoring, and reasoning. Where token-usage tracking is available, costs are recorded via the LangChain get_openai_callback context manager.

The data integration logic itself is implemented with PyDI (Python Data Integration), an in-house library that provides composable building blocks for blocking, matching, and fusion. PyDI exposes several blocker implementations (StandardBlocker, TokenBlocker, SortedNeighbourhoodBlocker, and EmbeddingBlocker backed by sentence-transformers), comparator types (StringComparator, NumericComparator, DateComparator), a RuleBasedMatcher with weighted decision functions, an MLBasedMatcher with scikit-learn classifiers, and a DataFusionEngine with configurable per-attribute fuser strategies.

Generated pipeline code is written to disk and executed as a standalone Python subprocess. This design ensures that the LLM-produced code runs in an isolated environment and that execution errors (import failures, type mismatches, runtime exceptions) are captured cleanly and fed back to the LLM for correction.

Ablation Design

The agent variants are designed as a cumulative ablation study. Rather than testing isolated features, each new agent tier retains all capabilities of its predecessor and adds exactly one new component or mode. This makes it possible to attribute changes in fusion accuracy, token usage, and runtime to a specific addition.

The progression is as follows:

  1. Agent I establishes the baseline: the LLM receives dataset profiles and an example pipeline, and must infer blocking, matching, and fusion strategies entirely on its own. Only rule-based matching is supported.
  2. Agent II adds the Blocking Tester and Matching Tester nodes. These evaluate multiple strategies against gold-standard test sets before pipeline generation, providing the LLM with pre-computed, empirically validated configurations. This isolates the effect of strategy selection from the LLM's code-generation ability.
  3. Agents III (RB) and IV (ML) introduce the Reasoning Node, which analyses evaluation failures and produces structured improvement suggestions before looping back to pipeline generation. Agent IV additionally switches from rule-based to ML-based matching, enabling us to compare matcher paradigms under otherwise identical conditions.
  4. Agents V (RB) and VI (ML) add the Cluster Analysis node and the Search Documentation Tool. The cluster analysis diagnoses over-matching after each pipeline run, while the documentation tool gives the LLM access to PyDI API documentation at generation time, reducing hallucinated function calls.

Because each tier is strictly additive, differences in evaluation metrics between adjacent tiers can be attributed to the newly introduced component with reasonable confidence. The main confound is the inherent non-determinism of LLM-generated code; we mitigate this by running each agent variant across all four use cases and reporting both per-domain and aggregate results.

Controlled Variables

To ensure a fair comparison across agent variants, several variables are held constant throughout the experiment suite.

Blocking and matching configurations. For Agents II through VI, the optimal blocking and matching strategies are determined once per use case by the tester nodes and then reused identically across all agent variants that support them. This means that differences in fusion accuracy between, say, Agent III and Agent V cannot be explained by different blocking keys or matching thresholds.

Exception: Agent VII

Agent VII introduces a dedicated normalization node that preprocesses attribute values before blocking and matching. Because normalization changes the surface form of the data, the blocking and matching configurations for this agent differ from the shared baseline. Results for Agent VII are therefore reported separately and should not be directly compared with the other variants on the controlled blocking/matching dimension.

LLM model and temperature. All agents use GPT 5.4 with temperature set to 0, ensuring deterministic decoding for a given prompt. The same model is used for schema matching, pipeline generation, evaluation code generation, and the reasoning node.

Example pipelines and prompts. The example integration pipeline and example evaluation script provided to the LLM are identical across all agent variants for a given matcher mode (rule-based or ML). When the matcher mode is switched to ML, a separate ML-specific example pipeline is used, but this example is again shared across all ML-mode agents.

Retry budgets. Pipeline and evaluation execution allows up to three retry attempts and the outer evaluation-reasoning loop runs for at most three iterations. These limits are consistent across all agents.

Gold-standard test sets. The entity-matching and fusion test sets are fixed per use case and shared across all runs. They are not regenerated between experiments.

Evaluation Protocol

Pipeline quality is measured through fusion accuracy, which captures how well the final fused output matches the gold-standard reference. The evaluation proceeds in two stages.

In the first stage, the agent generates an evaluation script that loads the fused output CSV and the gold-standard test set. The script aligns records between the two files using entity identifiers and computes per-column accuracy: for each attribute in the fused output, the fraction of records where the fused value matches the gold-standard value. Comparison is performed after basic normalization (lowercasing, whitespace stripping) to avoid penalising trivial formatting differences.

In the second stage, the per-column accuracies are aggregated into a single overall accuracy score (the mean across all evaluated columns). This overall accuracy is the primary metric used by the Evaluation Decision node to determine whether the pipeline should be refined further or whether the run should terminate. The termination threshold is set at 85% overall accuracy; if the agent has not reached this threshold after three evaluation-reasoning iterations, the run terminates and the best result is recorded.

In addition to accuracy, the evaluation infrastructure records structured per-record debugging events in JSONL format. Each event captures the entity identifiers, the expected and actual fused values, and the column-level match status. These events are consumed by the Reasoning Node in Agents III–VI to diagnose specific error patterns (false positives, false negatives, preprocessing mismatches) and to produce targeted improvement suggestions for the next pipeline iteration.

Separately from fusion accuracy, the Blocking Tester and Matching Tester nodes use Pairs Completeness (PC) and F1 score as their internal quality metrics. PC measures the recall of the blocking step (what fraction of true matches survive candidate generation), while F1 measures the balance of precision and recall for the matching step. These metrics guide strategy selection but are not directly used as the final evaluation criterion; the end-to-end fusion accuracy is the definitive measure of pipeline quality.

Use Case Analysis

Games

The Games use case is the most forgiving domain for the agent family. The three sources share several strong identity signals such as name, platform, developer, and releaseYear, and the strongest pipelines exploit this structure consistently. The main pattern across the snapshots is that exact-name blocking can occasionally work very well, but the more reliable runs move to semantic blocking and then improve further through either reasoning-guided rule calibration or pair-specific ML matching.

At the same time, the strong but unstable baseline snapshot shows that this domain can occasionally be solved even with simple rules when the naming conventions happen to line up.

Music

The Music use case is materially harder. The sources differ not only in naming conventions, but also in how they represent artists, track lists, duration, release dates, and release-country information. As a result, exact blocking on a single field is too brittle, and even the better pipelines need a mix of semantic blocking, release metadata, and list-aware comparison logic to reach acceptable accuracy.

Findings

Token Usage

The token analysis below is based on the node_activity.json logs for all 28 recorded AdaptationPipeline_* runs across the Games, Music, Books, and Restaurant use cases.

This chart provides the aggregate token-usage view that complements the detailed family breakdown in the table below.

At a family level, token usage grows almost monotonically with additional reflective nodes. The cheapest family is the baseline AdaptationPipeline rule-based notebook at roughly 52.6k tokens per run on average. Adding blocker and matcher testing raises this to 83.5k, reasoning pushes the rule-based and ML variants to roughly 132.1k and 149.7k, and the cluster-based ML variant reaches 218.2k. The heaviest notebooks are AdaptationPipeline_Reasoning_Helper_ml at 399.6k average tokens and AdaptationPipeline_with_new_Cluster rule-based at 763.9k, driven by repeated reasoning and report-generation loops.

Looking by use case, Books, Games, and Music are very close in average tracked token usage, all around 273k-275k per run, while Restaurants is lower at roughly 205.1k. This means the most important predictor of token usage is not the domain alone, but how many higher-level reasoning, investigation, and summarization nodes a notebook family introduces.

Anomalies

Several token patterns stand out and need interpretation rather than a literal reading.

  • Extreme token spike in cluster-based rule runs: AdaptationPipeline_with_new_Cluster_rule_based reaches 1,145,538 total tokens. The log shows five separate pipeline_adaption passes, two large evaluation_reasoning passes, three token-heavy cluster_analysis stages, and a final save_results step of 131,549 tokens. This is not a single oversized prompt; it is an accumulated effect from many refinement and reporting loops.
  • Zero-token tester nodes are not actually “free”: many runs record run_blocking_tester, run_matching_tester, cluster_analysis, normalization_node, or save_results with zero prompt/completion tokens. In the logs this usually corresponds to state reuse, artifact reuse, or purely local processing. The node still performs work, but not new LLM work.
  • The baseline understates total effort if read as “cheap overall”: the baseline family is cheap in tracked tokens because it delegates most of the hard work to the generated pipeline execution itself. Token cost and total system cost are therefore not interchangeable in this project.

Time Complexity

The runtime logs show a different story from the token logs. Across all 28 runs, the average duration contribution of execute_pipeline is roughly 1,373.7 seconds whenever that node appears, which is far above the average durations of pipeline_adaption at 158.8 seconds, evaluation_reasoning at 76.9 seconds, or evaluation_adaption at 56.3 seconds. In practice, the end-to-end time complexity is therefore dominated first by how often the system reruns the generated integration pipeline, and only second by how many reasoning prompts it issues.

The notebook families share the same basic scaling structure. With three sources, the pipeline must process three pairwise matching tasks, so candidate generation and matching already grow with the number of dataset pairs. Within each pair, semantic blocking, feature extraction, classifier search, clustering, and fusion each add another layer of work. The ML notebooks intensify this by running pair-specific feature extraction and GridSearchCV over multiple classifiers, while the cluster and reasoning variants multiply the cost further through iterative refinement. Empirically, the logs behave like iterations × pairwise execution cost rather than simply prompt size.

By use case, Games is the clear runtime outlier at roughly 3,540.0 seconds average per run, compared with 1,920.8 seconds for Books, 1,267.4 seconds for Music, and 1,159.4 seconds for Restaurants. This matches the Games logs, where semantic blocking configurations report candidate counts in the 409,504 to 929,573 range, so even strong blocking still leaves large local workloads for matching, clustering, and fusion.

Anomalies

The runtime traces contain several non-obvious outliers that are worth interpreting.

  • Very long runtime with relatively low token usage: AdaptationPipeline_blocking_matching_extension_Final_Blocker_Matcher_rule_based takes about 5,633.5 seconds but only records 60,128 tokens. The reason is that the run executes the generated pipeline four times, with each execute_pipeline call taking roughly 1,350-1,393 seconds. This is a local execution bottleneck, not an LLM bottleneck.
  • Games is the slowest domain without being the most token-heavy: the Games average runtime is far above the other use cases even though Games token usage is similar to Books and Music. The most plausible explanation, supported directly by the blocking summaries, is that the candidate sets remain much larger, so execution, clustering, and fusion dominate the wall clock.
  • Reasoning-helper ML pays large fixed overheads outside the core matcher: in AdaptationPipeline_Reasoning_Helper_ml, normalization_node alone consumes about 217.8, 184.0, and 181.0 seconds across its three appearances, while investigator_node consumes up to 365.9 seconds. The family is therefore slow partly because it adds expensive diagnostic infrastructure around the matching pipeline.
  • Zero-duration or near-zero tester nodes should be read carefully: some run_blocking_tester, run_matching_tester, and cluster_analysis entries show almost no time because the run reused precomputed state or produced a no-op pass. They are not evidence that these stages are universally constant-time; they are evidence that the specific logged execution path reused prior artifacts.
